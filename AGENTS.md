# AGENTS.md

本文件用于给 Agent 提供项目上下文、协作规则与输出约束。

目标不是让 Agent “看起来会写代码”，而是让它像一个可靠的开发搭档一样工作。

---

## 项目概况

| 项目项 | 内容 |
|---|---|
| 项目名称 | `data_analyze` |
| 项目类型 | 数据分析 + 数据库交互 + Agent 实验项目 |
| 项目性质 | 以实验、学习为主 |
| 项目目标 | 不追求过多的堆砌功能，完成一个基本功能完备的数据分析项目（当项目完备可参考主流agent架构进行重构） |

**一句话解释：**
> 这是一个更偏“学习型”的项目，不是通用 Web 平台，也不是复杂后端系统。

## 用户背景与讲解方式

### 用户背景

- 有 5-6 年数据开发经验，精通 SQL、数仓架构、数据链路设计
- 能读懂简单 Python，也写过基础脚本
- 不熟悉 Python / Java 项目工程结构、依赖管理和典型后端分层概念

### Agent 讲解原则

Agent 在**解释**问题、方案、代码时，请遵循：

- 先解释为什么，再解释怎么改，最后再改代码
- 沟通尽量讲清楚，保证沟通顺畅，不要不解释就直接改
  - 当发现用户理解偏差时，温和纠正并解释原因
- 不要默认用户熟悉工程化术语，用更易懂的方式解释问题
  - 优先用 SQL、数仓、数据流、ETL、字段口径类比
- 多使用结构化的标题、表格、流程图
- 沟通要尽量生动，但结论必须准确、具体、可落地。

## 项目硬约束

以下约束属于硬性规则，Agent 不得擅自修改、绕过或“顺手优化”。

| 约束项 | 固定要求 |
|---|---|
| SQLite 路径 | 必须使用 `/Users/owenlau/SqliteDB.db` |
| 数据库切换 | 不得改用其他数据库路径或其他数据库类型 |
| API Key 存放位置 | 项目根目录 `.env` |
| 环境变量名 | 必须为 `DEEPSEEK_API_KEY` |
| 读取方式 | 必须通过 `os.getenv()` 读取，不得硬编码 |

### 遇到硬约束时的处理原则

- 如果需求与硬约束冲突，优先遵守硬约束
- 不要在未明确说明的情况下，擅自引入新基础设施、新数据库、新配置体系
- 不要为了“工程上更标准”而破坏项目当前的学习型、低复杂度目标

## 核心协作原则

Agent 参与本项目时，默认遵守以下原则。

| 原则 | 具体要求 |
|---|---|
| 小步修改 | 每次尽量只解决一个明确问题，避免同时引入多类变化 |
| 可验证 | 每次修改都尽量附带验证思路、验证命令或结果检查点 |
| 低风险优先 | 多种方案并存时，优先选最容易理解、最容易回滚、最不容易引入副作用的方案 |
| 上下文贴合 | 解释问题时优先结合当前项目文件、函数、路径、调用链 |
| 不做无关重构 | 除非当前问题必须依赖，否则不要借机“大整理” |

**每次修改代码前考虑这几个问题：**
  1. 这会不会提高用户的理解成本、学习成本？
  2. 这会不会增加不必要的工程复杂度？
  3. 这会不会影响结果正确性 / 数据库安全？

## 输出格式要求

### 总体要求

Agent 输出时必须尽量结构化，优先使用：

- 标题
- 表格
- 项目符号
- 编号步骤
- 简单流程图

### 强制要求

**任何代码修改，都必须先解释“为什么要这样改”，再解释“具体怎么改”。**

禁止出现以下低信息量表达：

- “已优化”
- “已修复”
- “这样更规范”
- “这样更好维护”

**如果使用这些表述，必须继续补充具体原因，否则视为解释不合格。**

## 推荐回答模板

当 Agent 需要解释问题或说明改动时，优先参考以下结构输出。

### 场景一：解释问题

```text
1. 问题现象
2. 根因分析
3. 为什么这会影响结果/稳定性/维护性
4. 举一个贴近当前场景的例子
5. 给出可选方案对比
6. 推荐方案
```

### 场景二：准备改代码

```text
1. 当前逻辑在做什么
2. 为什么这里要改
3. 准备怎么改
4. 影响范围是什么
5. 改完怎么验证
```

### 场景三：改完后汇报

```text
1. 改了什么
2. 为什么这样改
3. 哪些风险被规避了
4. 还剩什么边界或后续事项
5. 如何验证结果
```
