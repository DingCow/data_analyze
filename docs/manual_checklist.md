# 手工验收样例

这份清单的目标不是替代自动测试，而是补一层“真实业务问题验收”。

自动测试更像检查“代码骨架有没有断”；
这份清单更像检查“用户实际提问时，整条链路是否还能正常工作”。

适用场景：
- 修改了 prompt
- 修改了 Agent 路由或调用顺序
- 修改了前端展示逻辑
- 修改了 SQL 生成策略
- 准备把当前版本继续往下开发前，先做一次人工验收

建议执行频率：
- 日常小改动：至少跑前 3 条
- 改动 Agent / prompt / 路由：5 条全部跑

---

## 执行方式

优先在 Web 端执行：

```bash
streamlit run app.py
```

如果只想快速验证链路，也可以用 CLI：

```bash
python3 main.py
```

每条问题都关注 4 个点：
- 是否成功返回结果，而不是直接报错
- 返回内容是否和问题类型匹配
- 简单查询是否直接出表格
- 复杂分析是否有文字结论，必要时有图表

---

## 样例 1：最简单查数

问题：

```text
users_info 表前 5 行数据是什么？
```

期望：
- 能返回结果
- 路由应更接近 simple 路径
- Web 端应更接近直接展示表格
- 不需要复杂分析结论

意义：
- 验证最基础的“自然语言 -> SQL -> 返回数据”链路是否可用

---

## 样例 2：聚合统计

问题：

```text
users_info 一共有多少条记录？
```

期望：
- 能返回 1 行聚合结果
- 不应返回大量明细
- 不需要复杂报告

意义：
- 验证聚合类问题是否还能稳定走 simple 路径

---

## 样例 3：时间趋势分析

问题：

```text
2024 年各月充电收入趋势如何？
```

期望：
- 能成功返回分析结果
- 路由应走 complex 路径
- 应有分月结果
- Web 端最好能出折线图
- 结论里应出现趋势描述，而不是只贴原始数据

意义：
- 验证最关键的复杂链路：Router -> Analysis -> SQL -> Analysis -> Report

---

## 样例 4：空结果场景

问题：

```text
2026 年各月充电收入趋势如何？
```

期望：
- 不报错
- 能明确告诉用户“无数据”或结果为空
- 不能编造趋势结论

意义：
- 验证空结果处理是否稳健
- 这类场景很容易暴露 prompt 幻觉或异常处理问题

---

## 样例 5：跨表或业务分析表达

问题：

```text
2024 年哪个城市的充电收入最高？请给出前 10 名并简单总结。
```

期望：
- 能返回排序结果
- 应有“前 10 名”这类明确限制
- 如果走 complex 路径，结论应和数据一致
- 不能只输出笼统描述

意义：
- 验证“排序 + TopN + 总结”这类常见分析问法是否可用

---

## 如何判定这次改动可以接受

最低通过标准：
- 5 条里至少 4 条成功
- 样例 3 必须成功
- 样例 4 不能编造结论
- 不能出现写库相关报错或异常 SQL 行为

如果失败，优先判断是哪一层问题：
- 一进来就报错：先看数据库连接或配置
- simple 问题走成复杂分析：先看 Router 分类
- 能查到数据但结论很差：先看 Analysis / Report
- SQL 被拦截但本应合法：先看只读校验规则

---

## 推荐搭配

每次人工验收前，先跑自动测试：

```bash
python3 -m unittest discover -s tests -v
```

建议顺序：
1. 先跑自动测试，确认骨架没断
2. 再跑这 5 条手工样例，确认真实体验没坏

这样你就同时覆盖了：
- 代码层回归
- 业务链路回归
