# Apple Workbench UI Design

## 1. 目标

本次改造的目标不是重做产品结构，而是在保留当前 Streamlit 页面布局逻辑的前提下，把 Web UI 提升到更接近苹果公司设计审美的专业工作台质感。

当前已确认的约束如下：

- 保留现有布局逻辑，不做明显重排
- 保留现有功能模块和交互链路
- 重点改造视觉层，不改业务逻辑
- 使用 `pencil` MCP 先产出设计稿，再回写到代码
- 原始 UI 样式必须先备份，保证可回退

一句话目标：

> 让当前页面从“有设计的 Streamlit 页面”，升级成“像 macOS 桌面分析工具的轻量工作台”。

---

## 2. 现状理解

当前页面骨架已经完整，主要由四个区域构成：

```text
顶部：标题 / 状态
  ↓
左侧：问题历史 / 过程说明
右侧：结果摘要 / 分析结论 / 图表表格
  ↓
底部：统一输入区
```

现有问题不在功能缺失，而在视觉语言还不够收敛：

- 背景、卡片、阴影和边框之间没有形成统一的系统层级
- 某些区域更像普通 Web 卡片，而不是桌面应用面板
- 结果区、过程区、输入区的视觉权重还有进一步拉开的空间
- 页面已经有“浅色分析工作台”方向，但距离苹果式的克制和精致还有差距

---

## 3. 设计方向

### 3.1 总体方向

采用“方案 A：轻量换肤”路线。

这意味着：

- 不改变页面整体结构
- 不引入新导航体系
- 不拆分现有内容模块
- 只通过背景、材质、留白、圆角、边界、字体层级来提升质感

### 3.2 视觉关键词

本次设计稿应围绕以下关键词展开：

- 安静
- 通透
- 柔和
- 克制
- 高级
- 桌面应用感

### 3.3 明确不做的事

- 不做营销官网风
- 不做明显炫光、强拟物、强彩色渐变
- 不做深色模式主导
- 不做高复杂度交互重构
- 不把页面改成全新的多栏产品结构

---

## 4. 视觉系统规则

### 4.1 背景

背景改造成更接近 macOS 桌面应用的轻雾面层次。

原则：

- 保留浅色底
- 允许非常轻的冷色雾面渐变
- 大面积背景不能太花
- 背景只负责提供环境感，不负责抢焦点

预期效果：

- 页面更像系统窗口所在的桌面环境
- 中间面板更容易被感知为前景层

### 4.2 面板

现有多个内容区会统一为“轻玻璃感面板”语言。

原则：

- 使用半透明浅色面
- 保留很轻的边框
- 阴影更柔和、更贴面
- 不做厚重投影和强描边

面板之间的层次应体现：

- 顶部信息层最轻
- 左侧历史层次次要
- 右侧结果层最聚焦
- 底部输入区像桌面工具的操作托盘

### 4.3 圆角

统一为更大的连续圆角体系，减少当前零散的圆角感知。

目标：

- 让整个页面看起来像来自同一个设计系统
- 避免某些卡片过硬、某些输入框过圆导致的节奏不一致

### 4.4 字体与层级

保留现有“标题 + 说明 + 正文 + 辅助信息”的结构，但重新梳理层级。

规则：

- 标题更像桌面应用中的主标题
- 辅助说明更轻、更短、更安静
- 正文和分析内容优先保证清晰可读
- 视觉层级靠字号、字重、颜色明度差来拉开，而不是靠花哨装饰

### 4.5 颜色

主色仍可保留蓝色系，但要更克制，减少传统 Web CTA 味道。

原则：

- 蓝色用于状态、主按钮、重点信息
- 中性色承担大部分界面面积
- 成功 / 异常状态色只在必要处出现
- 色彩数量控制，避免页面显得杂

---

## 5. 分区改造设计

### 5.1 顶部品牌区

当前顶部已经承担“标题 + 状态”的职责，不改结构，只改气质。

改造重点：

- 标题更像桌面应用标题栏下的主标题
- 状态卡片更轻，更像系统状态胶囊
- 整体间距更克制，减少块状感

预期结果：

- 顶部更像“工作区入口”，而不是传统网页 Banner

### 5.2 左侧问题历史区

左侧区域继续保留为次级信息区。

改造重点：

- 面板存在感降低，但可读性提升
- 历史记录像桌面应用的活动流容器
- 标签、说明、过程文本更轻，不和主结果区争视觉中心

预期结果：

- 左侧“看得到，但不抢主画面”

### 5.3 右侧结果工作区

右侧是本次视觉改造的核心区域。

改造重点：

- 结果摘要区更像主内容窗口
- 图表、表格、分析结论之间统一间距和层级
- 摘要卡和 insight 卡收敛成同一套面板语言
- 通过更好的留白提升专业感

预期结果：

- 用户第一眼自然落在结果区
- 数据结论看起来更像“正式输出”

### 5.4 输入区

输入区保留当前底部统一入口的交互方式。

改造重点：

- 输入框像 macOS 桌面工具里的主输入控件
- 主按钮减少传统网页 CTA 的营销感
- 整个输入面板更像底部操作托盘

预期结果：

- 用户感知为“专业工具的操作入口”，而不是单独一块表单

## 6. 备份策略

由于用户明确要求保留原始 UI 样式，本次实施前必须先做备份。

备份方案：

1. 复制当前 `app.py` 中的样式实现，保留一份原始版本
2. 备份方式优先采用同目录副本文件，文件名体现 `backup` 与日期
3. 若后续样式调整失败，可以直接对照回滚

推荐备份文件：

- `app.py` 的完整备份副本

说明：

- 本项目当前目录不是 Git 仓库，无法依赖 git commit 保存设计节点
- 因此本次回退能力主要依赖文件级备份

---

## 7. Pencil 使用方案

本次实现必须使用 `pencil` MCP。

使用方式如下：

1. 在 `pencil` 中创建当前页面的改造目标稿
2. 设计稿以“单页面工作台”形式表达
3. 重点展示以下区域：
   - 顶部标题与状态
   - 左侧历史面板
   - 右侧结果面板
   - 底部输入区
4. 设计稿不追求重做布局，而是突出材质、边界、色彩、字体层级
5. 设计稿确认后，再把样式映射回 `app.py` 的 CSS

这样做的原因是：

- 先在设计层确认“苹果感”是否成立
- 避免直接在 Streamlit CSS 上盲调
- 降低返工成本

---

## 8. 实施边界

本次实现只应修改与 UI 展示有关的部分。

允许改动：

- `app.py` 中的 CSS
- 少量与页面展示结构相关的 HTML 片段

不应改动：

- `src/` 下业务逻辑
- LLM、路由、SQL、分析流程
- 数据库配置与路径
- `.env` 和 API Key 读取逻辑

---

## 9. 验证方案

### 9.1 视觉验证

主要看以下问题是否达成：

- 页面是否明显更像桌面应用工作台
- 是否有统一的玻璃面板语言
- 结果区是否更聚焦
- 左侧和输入区是否更安静
- 是否保留了原页面结构认知

### 9.2 功能验证

启动方式：

```bash
streamlit run app.py
```

验证要点：

- 页面能正常打开
- 输入和提交功能正常
- 历史区仍能显示
- 结果区、图表、表格仍能显示
- 清空对话等原有操作不受影响

### 9.3 回归验证

优先执行已有自动测试：

```bash
python3 -m unittest discover -s tests -v
```

然后执行文档中的手工检查：

- 参考 `docs/manual_checklist.md`

## 10. 风险与控制

### 风险 1：苹果风做成普通浅色 Web 风

控制方式：

- 先出 `pencil` 设计稿
- 先确认材质和层次，再写 CSS

### 风险 2：只改颜色，没有改出系统感

控制方式：

- 同时调整背景、边界、圆角、阴影、文字层级
- 不把“苹果风”误解成单纯蓝白配色

### 风险 3：Streamlit 原生 DOM 限制导致样式不稳定

控制方式：

- 优先复用现有选择器结构
- 保持改造范围集中在现有已经控制住的区域
- 避免依赖脆弱的深层结构调整

### 风险 4：没有 Git，回退成本提高

控制方式：

- 先做文件级备份
- 小步修改
- 每轮改完都做页面验证

---

## 11. 推荐实施结论

推荐实施顺序：

1. 先备份当前 `app.py`
2. 用 `pencil` 产出苹果化工作台目标稿
3. 根据目标稿重写 `app.py` 的视觉样式
4. 启动 Streamlit 做页面核验
5. 跑自动测试和手工验收清单

本方案的核心不是“把页面变得更花”，而是让它在完全保留当前工作方式的前提下，看起来更像一个真正成熟的 macOS 分析工具。
